# Remediation System

What happens when a learner fails. This is the platform's most intricate subsystem â€” and the one with the strictest architectural rule:

> **All remediation decisions are made in exactly one place: `src/core/remediation_policy.decide_remediation`.** Routes, services, the chat agent, and hooks read its verdict; none of them decide anything themselves. If you change remediation behavior anywhere else, you are reintroducing the bug this design fixed (four independent mechanisms each deciding "this learner needs help" for the same failure, sometimes contradicting each other).

## The decision

`decide_remediation` fuses two deterministic engines into one `RemediationDecision`:

**1. State machine verdict** (`state_machine.handle_assessment_result`):
- Score â‰Ą 80% (`PASS_THRESHOLD`) â†’ `passed`.
- **Case 1** â€” failed *bypass* attempt (veteran/intermediate fast-track) â†’ `bypass_locked`, `lock_bypass=True`. The full learning path becomes mandatory; already-completed courses are excluded (`get_mandatory_courses`). Bypass stays chat-only â€” the quiz UI has no bypass affordance; the agent hook enforces the lockout there.
- **Case 2** â€” standard failure â†’ `gap_review`; the learner reviews weak areas and retakes.

**2. Luck-elimination verdict** (`LuckEliminationEngine.evaluate_user_progression`), computed over the all-time `error_retention_matrix` with this attempt's wrong answers folded in:
- Any concept failed â‰Ą 2Ă— (`LUCK_FAILURE_THRESHOLD`) is *flagged*.
- â‰Ą 3 flagged concepts (`core_drift_concept_count`) â†’ `FORCE_MANDATORY_LEARNING_PATH` â€” the learner is guessing at scale.
- 1â€“2 flagged â†’ `SPAWN_GAP_REVIEW`.
- 0 â†’ continue normally.

The fused decision fields: `next_state`, `lock_bypass`, `luck_action`, `flagged_concepts`, `spawn_gap_review`, `spawn_remedial_course` (true **only** for a failed `final_assessment`), and a human-readable `reason` that travels to the UI.

## What the caller does with it

`POST /api/quiz/evaluate` (and, for chat flows, `routing_service.handle_user_assessment_failure` â€” a thin persistence wrapper) executes the verdict:

- `lock_bypass` â†’ persist `bypass_locked`/`bypass_attempts`, compute mandatory courses.
- `spawn_gap_review` â†’ `generate_gap_review` builds the review payload.
- `spawn_remedial_course` â†’ `generate_remedial_course` (LLM) builds a targeted mini-course.

## Gap review + HLR filtering

`generate_gap_review` surfaces flagged concepts as exercises â€” but first filters by memory decay: for each concept with a `mastery_vectors` entry it computes `calculate_hlr_retention` (`p = 2^(-Î”t/half_life)`); concepts with retention â‰Ą 0.6 (`hlr_retention_threshold`) were reviewed recently and are scheduled for later instead of dumped on the learner again. Concepts with no vector fall back to immediate inclusion. Exercises reuse stored `concept_diagnoses` (the real misconception text) when available, instead of a generic "review the concept" template. In the UI each concept gets a **Start Targeted Retry** button â†’ `POST /api/quiz/gap-review/retry` â†’ a 3-question quiz seeded with the diagnosis text.

## Remedial courses

Generated by `generate_remedial_course` (Gemini, `generate_remedial_course` template) from the learner's actual wrong answers. Two things happen on generation:

1. **Diagnosis persistence** â€” the prompt asks for a per-wrong-answer root-cause `misconception`; these are zipped against the incorrect answers and stored in `progress["concept_diagnoses"]` keyed by concept tag. Gap review and reflection prompts reuse them, so the "why you got this wrong" analysis is written once and used everywhere.
2. **Accumulation cap** â€” at most 2 pending (uncompleted) remedial courses per `source_course_id` (`remedial_course_cap`). A third failure **merges** the new gap topics into the most recent pending course and regenerates its content (tracked via `regenerated_count`) instead of stacking endless courses.

Remedial courses live in the user's progress record and are merged into path views at read time with `is_remedial: true` â€” the dashboard renders the "TARGETED REVIEW" badge.

## Metacognitive reflection

On any wrong answer mid-quiz, the frontend calls `POST /api/quiz/reflection`; `generate_reflection_prompt` returns a Socratic prompt (prefilled with the stored diagnosis when one exists) rendered inline under the feedback.
