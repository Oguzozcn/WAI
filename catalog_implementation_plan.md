# Knowledge Base Vault: Catalog Feature Implementation Plan

## Goal Description
The objective of this plan is to design and implement the Knowledge Base Vault's cataloging and storage architecture for the MVP. This implementation will cleanly separate raw manager uploads (inputs) from the curriculum builder's generated learning paths (outputs). 

This clear separation achieves four primary goals:
1. **Prompt Engineering & Testing:** Storing the direct output of the `curriculum_builder` alongside its specific input file allows for easy input vs. output comparison, which is critical for refining and optimizing LLM prompts.
2. **Dashboard Selection:** Persisting the generated standard Learning Paths allows the frontend Dashboard to fetch and display a catalog of available courses, enabling users to choose between different learning paths.
3. **Personalized Gap Learning:** Creates a dedicated user-specific storage area for `knowledge_gap_learning_path` files. These are triggered only when a user fails a final assessment and needs remediation.
4. **KPI Tracking:** The folder structure is optimized so the `department_reporter` agent can seamlessly parse who is taking which learning path (standard vs. gap) to aggregate reporting KPIs.

## Decisions Lock

| Decision | Answer |
|----------|--------|
| **Inputs per Path** | **Single Input (1:1) for MVP.** Support for multiple inputs merging into one path will be a future feature. |
| **Version Control** | Handled by the `kb_validator` agent. Old learning paths will be renamed by appending `_previous_v1` or `_old_v1` instead of overwriting. |
| **Gap Path Lifecycle & Final Assessment** | After completing a gap path, user takes a **NEW Final Assessment** covering both the failed modules and earlier sections (to prove it wasn't a luck choice). If they fail again, a 2nd gap path is generated on current failed modules. **Max 3 fails** before requiring **Human-in-the-loop (Manager intervention)**. |
| **Unofficial Paths** | Users can create their own custom learning paths, stored separately from official department paths. |
| **Path Assignments & Dashboard** | Managers can assign multiple official learning paths to users. The manager progress dashboard will track and display user progress across both official (assigned) and unofficial (self-created) paths. |
| **Knowledge Vault UI & RBAC** | The UI will feature "Learning Materials" (inputs) and "Learning Paths" (outputs) folders. Managers see all materials/paths. Users see assigned official paths and their own unofficial paths, with restricted access to raw learning materials. |

## Proposed Architecture & Changes

### 1. Directory Structure Update

We will update the persistence layer to manage the new folder hierarchy, keeping everything strictly within the `department_id` namespace.

#### [MODIFY] `shared/persistence.py`
Add logic to initialize and enforce the following sub-directory structure inside `data/knowledge_base/{department_id}/`:

```text
data/
└── knowledge_base/
    └── {department_id}/
        ├── catalog/
        │   ├── inputs/                  ← Raw uploaded documents from Managers (.txt, .md)
        │   │   ├── doc_vertex_ai.md
        │   │   └── doc_sales_v1.txt
        │   │
        │   ├── standard_paths/          ← Generated by `curriculum_builder` (The Outputs)
        │   │   ├── lp_vertex_ai.json    ← Baseline path for all department users
        │   │   └── lp_sales_v1.json
        │   │
        │   ├── unofficial_paths/        ← User-generated custom learning paths
        │   │   └── {user_id}/           ← e.g., emp_001/
        │   │       └── custom_lp_python.json
        │   │
        │   └── gap_paths/               ← User-specific paths after failing final assessment
        │       ├── {user_id}/           ← e.g., emp_001/
        │       │   └── gap_lp_vertex_ai.json
        │       └── {user_id_2}/         
        │           └── gap_lp_sales_v1.json
```

*Note: By keeping `gap_paths` structured by `{user_id}`, the `department_reporter` can easily iterate through `data/knowledge_base/{department_id}/catalog/gap_paths/` to see exactly which users are struggling and what gap paths they are currently assigned.*

### 2. File Upload & Baseline Generation Flow (Manager POV)

#### [MODIFY] `tools/curriculum_tools.py` & `kb_validator`
- Update the asynchronous document processing flow:
  1. Save the raw uploaded file into `catalog/inputs/`.
  2. The `kb_validator` agent checks for existing versions of the document/path. If an older version exists, it renames the old path by appending `_previous_v1` or `_old_v1`.
  3. Invoke the `curriculum_builder` agent to generate the baseline learning path.
  4. Save the resulting generated JSON into `catalog/standard_paths/`.
  5. Ensure the generated JSON includes metadata mapping back to the input file (e.g., `source_input_files: ["doc_vertex_ai.md"]`) to help with prompt testing. (MVP is 1:1, but array format future-proofs it).

#### [MODIFY] `server.py`
- Expose a new endpoint (e.g., `GET /api/kb/catalog`) to serve the list of available paths from `catalog/standard_paths/`. This allows the frontend Dashboard to display a selection of Learning Paths for the user.

### 3. Gap Path Generation Flow (User POV)

#### [MODIFY] `tools/routing_tools.py` & `shared/state_machine.py`
- Update `handle_assessment_failure()` and state machine logic:
  - **First Failure:** Trigger the `curriculum_builder` to generate a specialized path focusing *only* on the failed modules. Save this output to `catalog/gap_paths/{user_id}/gap_{original_path_name}.json` and update user progress to point to it.
  - **Gap Path Completion:** Upon completing the gap path, the user is given a **NEW Final Assessment** covering the newly learned modules and previously passed ones (to eliminate luck choices).
  - **Second Failure:** If the new Final Assessment is failed, generate a second gap path based on the current failed modules.
  - **Third Failure (3-Strikes):** If failed a 3rd time, lock the assessment and require **Human Intervention (Manager Review)**.

### 4. Department Reporter Optimizations

#### [MODIFY] `tools/reporting_tools.py`
- Update `synthesize_department_kpi()` to parse the new structure:
  - Read `catalog/standard_paths/` to inventory the baseline paths available.
  - Read `catalog/unofficial_paths/` to inventory the self-created paths per user.
  - Read `catalog/gap_paths/` to count how many users are on remediation paths (a critical KPI for identifying difficult courses or poorly designed content).
  - Cross-reference with `user_progress/` to report exactly which users are enrolled in which paths, separating official assigned progress from unofficial progress.

### 5. Learning Path Assignment & Progress Tracking

#### [MODIFY] `shared/models.py` & `server.py`
- Update the `UserProgress` schema to support multiple concurrent enrollments, distinguishing between official and unofficial assignments:
  ```json
  {
    "assigned_paths": ["lp_vertex_ai", "lp_sales_v1"],
    "unofficial_paths": ["custom_lp_python"],
    "progress": {
       "lp_vertex_ai": { "status": "COURSE_IN_PROGRESS", "completed_modules": 2 },
       "custom_lp_python": { "status": "ENROLLED", "completed_modules": 0 }
    }
  }
  ```
- **Manager Dashboard APIs**: Create endpoints for managers to assign official learning paths to users, updating the `assigned_paths` list. Provide endpoints that aggregate progress across both `assigned_paths` and `unofficial_paths` for the Management Dashboard.

### 6. Frontend Knowledge Vault UI & Role-Based Access

#### [MODIFY] `knowledge-vault.html` & `server.py`
- **Google Stitch UI Integration:** Integrate the provided UI designs from the `ui_exports` directory. Specifically, utilize the generated assets and code for `learning_materials_knowledge_vault` and `learning_paths_knowledge_vault` to build out the frontend views for these two folders.
- **UI Structure:** Under the "All Materials" view in the Knowledge Vault, implement a "Folders" section containing two main directories:
  1. **Learning Materials:** Displays all uploaded input files and DTPs (e.g., `sample_dtp.json`, `vertex_ai_dtp.json`). This maps directly to the `catalog/inputs/` backend directory.
  2. **Learning Paths:** Displays all generated learning paths (e.g., "Gemini API in Vertex AI", "Cloud Computing Intro"). This aggregates data from `catalog/standard_paths/` and `catalog/unofficial_paths/`.
     - **Direct Navigation:** Clicking the "Learning Path" link in the main site navigation routes the user directly to this specific "Learning Paths" folder within the Knowledge Vault.
     - **Progress & Actions:** Each learning path listed in this folder will display an integrated **Progress Bar**. Users can choose which path to "Start" or "Continue" directly from this interface.

- **Role-Based Access Control (RBAC):**
  - **Management View:** Managers have full access to view both the "Learning Materials" and "Learning Paths" folders across the department.
  - **User View:** Users have restricted access. The "Learning Materials" folder is hidden or restricted. In the "Learning Paths" folder, users will only see official paths that are available to them, alongside their own self-created unofficial paths.

- Verify the generated learning path saves to `catalog/standard_paths/` and contains the `source_input_files` metadata.
- Simulate a final assessment failure; verify the gap path saves to `catalog/gap_paths/emp_001/` and the user's progress state points to it.
- Run `department_reporter` tool and verify it correctly counts users on standard paths vs gap paths.

- Developer checks the local filesystem to ensure `inputs/` and `standard_paths/` are populated correctly for input vs. output testing.
- Verify the Dashboard UI fetches and displays the new standard path as an option for users.
