Architectural Assessment and Professional Reorganization Blueprint for the WisdomAI Platform
The architectural assessment of the WisdomAI Transition Execution AI Platform (TEAP) reveals a system with highly specialized functional capabilities, including dynamic curriculum generation, adaptive routing, and namespace-isolated corporate reporting1. However, in its current prototype stage, the directory structure and code organization present significant architectural coupling that will impede enterprise scalability, team-based development, and production deployment1.
This report provides a structural critique of the current layout, defines a refactoring strategy based on FastAPI best practices, aligns the agentic runtime with the Google Antigravity Software Development Kit (SDK) and Agent Development Kit (ADK) specifications, and outlines an enterprise-grade deployment and evaluation framework.
Structural Critique of the Current Codebase Hierarchy
The legacy directory structure of the WisdomAI prototype utilizes a flat, co-located layout that mixes web presentation layers, core business logic, agentic execution environments, and local file-based databases within a single root folder1. This design creates immediate operational challenges, as configurations, execution scripts, and static templates compete for the same namespace, violating standard software packaging boundaries2.
WAI-main/ ├── server.py ← FastAPI application entry point ├── requirements.txt ← Python dependencies ├── scope_project.md ← Master architecture and scope document ├── verify_agents.py ← Verification script for agent checking
A primary structural anti-pattern is the location of the WAI_agent/data/ folder1. Co-locating runtime-generated JSON files, uploaded corporate training documents, and database seeding scripts directly inside the source code tree is a critical security vulnerability1. In a continuous integration and deployment pipeline, this configuration risks leaking corporate data, department-scoped statistics, or personally identifiable information (PII) directly into git version control repositories4.
Standard git-ignore patterns must be utilized to isolate the entire data namespace from the repository codebase4.
Furthermore, grouping the backend agentic routines into generic subfolders like sub_agents/, tools/, and shared/ under WAI_agent/ introduces a highly coupled monolithic design1. Under this organization, any modifications to a specific capability—such as altering the quiz scoring logic—require navigating changes across multiple decoupled files, including quiz_tools.py, models.py, and agent.py1.

	Refactoring the Web and Data Layer using FastAPI Best Practices
To transition WisdomAI into an enterprise-ready API, the web server layer must be completely refactored. The legacy application relies on a single server.py to initialize the FastAPI app, define global dependencies, and register endpoints across entirely separate business domains1. As additional capabilities are introduced, this centralized structure becomes a bottleneck for development and testing9.
Domain-Driven Modular Monolith Architecture
Following industry-standard FastAPI conventions, the codebase should transition to a domain-driven modular structure, often referred to as the Netflix Dispatch pattern9. Rather than grouping files globally by technical type, the application should partition its core capabilities into self-contained domain directories inside a unified /src folder9.
For WisdomAI, these boundaries map directly to its primary functional requirements1:
* Curriculum Domain (src/curriculum): Governs multi-day path planning, training ingestion, and content generation1.
* Coaching Domain (src/coaching): Coordinates user quizzes, active evaluations, and cognitive review cycles1.
* Validation Domain (src/validation): Detects contradictions and validates consistency across uploaded documents1.
* Reporting Domain (src/reporting): Aggregates department-level KPI metrics, scrubbing PII prior to summary drafting1.
* User Management Domain (src/users): Tracks readiness scores and state progressions from enrolled to certified1.
Each domain package houses its own endpoints (router.py), validation metrics (schemas.py), internal database mappings (models.py), and operational handlers (service.py)9. When a domain must interact with another module, it executes an explicit, fully qualified package import, ensuring clear trace-ability and preventing circular dependencies2.
Event Loop Preservation and Non-Blocking File I/O
FastAPI achieves high performance by executing asynchronous endpoints (defined via async def) directly on the single-threaded Python event loop9. When an endpoint is declared as asynchronous, the framework expects that all internal network, database, or disk operations are completely non-blocking and invoked with await9. Conversely, if a route is defined as synchronous (using def), FastAPI offloads the entire execution path to an external thread pool, preventing it from halting the event loop9.
The legacy persistence layer in persistence.py uses blocking local disk operations via the DepartmentScopedStore1. If these blocking calls are invoked directly within asynchronous endpoints, the single-threaded event loop will freeze, preventing the server from accepting or processing any parallel incoming network requests9.
To prevent event loop starvation, the application must offload blocking file reads and writes to a thread pool using Starlette's run_in_threadpool utility, or utilize fully asynchronous file system libraries9. For long-running, CPU-bound operations—such as parsing training manuals or generating complex multi-day agendas—the application should offload the work entirely to an asynchronous task queue like Celery, Arq, or RQ, rather than relying on FastAPI's in-process BackgroundTasks1.
Serialization and Schema Optimization
The current serialization layout can be optimized to prevent double-instantiation performance penalties9. In many FastAPI architectures, endpoints manually instantiate Pydantic model objects and return them directly9.
However, if a route decorator defines an explicit response_model, the framework is forced to validate the outgoing data twice: first when the developer instantiates the model within the endpoint, and second when FastAPI's internal jsonable_encoder serializes it against the decorator schema9.
The optimized pattern requires database service layers to yield raw dictionaries or mapping objects, allowing the framework to perform validation and serialization in a single pass9.
To ensure structural consistency across all domains, a global custom base model should be implemented, standardizing common data patterns9:






Python
from datetime import datetime
from typing import Any
from zoneinfo import ZoneInfo
from fastapi.encoders import jsonable_encoder
from pydantic import BaseModel, ConfigDict, field_serializer

class CustomModel(BaseModel):
   model_config = ConfigDict(populate_by_name=True)

    @field_serializer("*", when_used="json", check_fields=False)
   def _serialize_datetimes(self, value: Any) -> Any:
       if isinstance(value, datetime):
           if value.tzinfo is None:
               value = value.replace(tzinfo=ZoneInfo("UTC"))
           return value.strftime("%Y-%m-%dT%H:%M:%S%z")
       return value

   def serializable_dict(self, **kwargs):
       """Yields a clean dictionary containing only JSON-serializable values."""
       default_dict = self.model_dump()
       return jsonable_encoder(default_dict)

Furthermore, raising custom ValueErrors inside field-level validation schemas (utilizing the @field_validator decorator) allows the framework to automatically catch validation anomalies9. FastAPI intercepts these errors and formats them into standard HTTP    validation responses, removing the need for boilerplate try-except blocks inside API controllers9.
Relational Database and Database Migrations
Although the MVP currently uses JSON file databases, the architecture documentation outlines a migration path toward BigQuery or Google Cloud Storage1. To support this transition, the database schema layer must be structured from day zero using explicit naming conventions and managed schema migrations via Alembic9.
SQLAlchemy metadata should be configured with a standardized index and key naming convention rather than relying on database-specific default constraints9:






Python
from sqlalchemy import MetaData

POSTGRES_INDEXES_NAMING_CONVENTION = {
   "ix": "%(column_0_label)s_idx",
   "uq": "%(table_name)s_%(column_0_name)s_key",
   "ck": "%(table_name)s_%(constraint_name)s_check",
   "fk": "%(table_name)s_%(column_0_name)s_fkey",
   "pk": "%(table_name)s_pkey",
}

metadata = MetaData(naming_convention=POSTGRES_INDEXES_NAMING_CONVENTION)

To maintain architectural consistency as relational tables are defined, the schema definitions must adhere to strict structural constraints9:
* Table Formatting: Table names must be singular, formatted in lowercase snake_case, and prefixed by their parent domain module (e.g., coaching_quiz, coaching_question, reporting_department_summary)9.
* Field Naming Suffixes: Datetime fields must utilize an _at suffix (such as completed_at), and date fields must use a _date suffix (such as reporting_date)9.
* Migration Integrity: All database migrations managed via Alembic must be designed to be completely static and reversible9. If dynamic data hydration is required, only the underlying table contents may be dynamic, never the structural schema9. The migration template inside alembic.ini should utilize a readable naming convention:

file_template = %%(year)d-%%(month).2d-%%(day).2d_%%(slug)s

Aligning the Agent Framework with the Google Antigravity SDK and ADK Guidelines
The MVP implementation of WisdomAI's AI orchestration layer is powered by the Google Antigravity SDK using the gemini-3.5-flash model1. However, the physical placement of agents, tools, and shared files within the WAI_agent/ directory is misaligned with the declarative standards established in the SDK's v2.0+ engine1.
Decoupling High-Level Agent Behaviors from Web Execution
The Google Antigravity SDK is designed around a decoupled, three-layer model that separates high-level agentic behaviors from low-level communication protocols14.
* Layer 1 (The Agent): Serves as the primary declarative configuration entry point, exposing an asynchronous context manager that handles binary initialization, tool registration, and global safety policy compilation14.
* Layer 2 (The Conversation): Manages the stateful session context, accumulating run logs, token usage metrics, and historical steps14.
* Layer 3 (The Connection): Controls transport-level mechanics14. For local executions, it establishes a WebSocket connection with a Go-based local harness, serializing commands via highly structured protobuf messages14.
The orchestrator script in the legacy prototype (WAI_agent/agent.py) mixes API route processing with agent state tracking1. To resolve this, the system should configure a dedicated Agent session for each user interaction using the SDK's native asynchronous context manager, ensuring clean execution sandboxes and resource isolation14.
Declarative Skills over Imperative Prompts
The current prototype packages agent logic within standard Python classes in the sub_agents/ directory1. Under the Google ADK and Antigravity specifications, this imperative approach is deprecated in favor of a declarative skill-based architecture17.
By encapsulating prompts, rules, and reference materials into standardized directory-based Agent Skills conforming to the agentskills.io specification, the codebase achieves separation of concerns10.
A directory-based skill separates instructions and resources into a predictable folder layout11:
.agents/ └── skills/ ├── curriculum-builder/ │ ├── SKILL.md ← YAML frontmatter and behavioral guidelines │ └── references/ │ └── curriculum_standards.md ← Structural constraints for learning paths
Transitioning the curriculum_builder, knowledge_coach, and kb_validator into file-based skills enables the runtime to load domain-specific prompts on-demand1. The main orchestrator uses the SkillToolset class to dynamically search, load, and run skills based on user intent12.
This progressive disclosure pattern avoids passing large blocks of instructions to the LLM on every turn, reducing baseline context usage by up to   , minimizing latency, and optimizing token consumption12.
Furthermore, Antigravity v2.0+ enforces project isolation; the engine no longer scans global directories for active commands or custom workflows15. To compile and load skills properly, all custom directories must be placed within a project-local configuration folder, specifically .agents/skills/15.
The configuration of the workspace context should utilize the strict folder groupings defined in the SDK guidelines18:
* Rules (In .agents/rules/): Always-active system instructions that are globally appended to every conversation in the active project directory18.
* Skills (In .agents/skills/): On-demand operational capabilities loaded dynamically when the user's prompt matches the skill's description field12.
* Workflows (In .agents/workflows/): Repeatable, multi-step prompt sequences triggered using custom / commands18.
Multi-Agent Orchestration via Agent-to-Agent (A2A) Protocols
WisdomAI uses a central orchestrator (agent.py) that delegates work to five specialized sub-agents1. ADK handles this hierarchy natively using a structured parent-subagent tree19. This topology limits conversation transfer pathways, making multi-turn dialogs more predictable and easier to debug19.
To support complex multi-step workflows—such as generating a lesson path and immediately drafting a corresponding quiz—the system can leverage ADK's built-in workflow agents, including SequentialAgent, LoopAgent, and ParallelAgent19.
Furthermore, Python ADK agents natively support the Agent-to-Agent (A2A) service model8. When an agent is initialized, the framework automatically mounts JSON-RPC endpoints that serve standard A2A protocols, allowing agents to coordinate as independent microservices8. This allows the supervisor and sub-agents to run as separate processes on different ports, isolating context windows and preventing the context-window traps common in single-agent architectures8.
Dynamic Hook Interception and Runtime Policies
The "Luck Elimination" mechanism in WisdomAI blocks user progress if the student fails a specific concept twice1. In the prototype, this is implemented as an external Python script (luck_elimination.py)1.
Under the Antigravity SDK, this rule should be refactored into a declarative safety policy executed at the hook layer14. The SDK's hook system allows developers to intercept, inspect, and modify agent decisions across strict execution phases14.
To enforce the "Luck Elimination" rule, the platform should register a PreToolCallDecideHook14:
1. When the agent attempts to trigger a routing tool (e.g., routing_tools.py) to fast-track a user, the hook intercepts the request1.
2. The hook evaluates the user's error_retention_matrix within the SessionContext1.
3. If the user's error count for the current concept satisfies   , the hook denies execution, returning allow=False1.
4. The hook injects a denial message back to the model, forcing it to bypass the fast-track path and route the user to remedial lessons1.
The declarative configuration of these guardrails should utilize the priority-based evaluation model built into the SDK's policy engine14. This enforces safety rules at the runtime level, ensuring that critical data policies are evaluated before any tools are allowed to execute14.
RAG Optimization with Managed Recipes
The legacy prototype implements custom document processing in curriculum_tools.py, using simple heuristic splitting to chunk raw files and generate lessons1. To improve retrieval accuracy and reduce custom code maintenance, the platform should adopt ADK's native RAG (Retrieval-Augmented Generation) patterns8.
Developers can integrate one of two standard RAG recipes8:
   * Vector Search Integration (rag-vector-search): Connects the platform to Vertex AI Vector Search 2.0, utilizing managed embedding and similarity search pipelines8.
   * Agent Search Integration (rag-agent-search): Uses Agent Platform Search (Discovery Engine) paired with a managed Google Cloud Storage (GCS) Data Connector8. Users can drop raw manuals directly into a storage bucket, and the managed pipeline automatically handles parsing, document chunking, and metadata indexing, removing the need for manual file parsing code1.

Enterprise Development, Evaluation, and Deployment Lifecycles
To ensure reliability during architectural transitions, the WisdomAI platform must adopt professional development, evaluation, and deployment processes8.
Virtual Environments and Dependencies
Modern Python development requires virtual environment isolation and locked dependency resolution to guarantee environment parity across local, staging, and production environments3.
Using the uv package manager, the platform initializes its environment and manages active packages via lockfiles8. This approach ensures that all dependencies are explicitly pinned and resolved without version conflicts during build steps3.
Continuous Code Quality and Linting
To maintain code quality across the codebase, standard formatting tools should be automated5.
The platform implements Ruff to manage linting, formatting, and import organization9. Running Ruff checks via the command-line interface ensures compliance with PEP 8 standards3:






Bash
# Analyze, format, and verify the codebase layout using Ruff
agents-cli lint

This automated verification step prevents common syntax errors, dead imports, and formatting inconsistencies from entering the shared repository2.

Enterprise Deployment and Monitoring
The Antigravity CLI separates cloud resource configuration from application code deployments8. This allows infrastructure changes to be managed independently of core application updates8.
      * Infrastructure (agents-cli infra): Provisions necessary GCP resources—such as service accounts, IAM policies, and GCS storage buckets—using Terraform8.
      * Deployment (agents-cli deploy): Packages the application into a Docker container, pushes it to a private container registry, and deploys it to the target runtime, such as Google Cloud Run or Google Kubernetes Engine (GKE)8.
Once deployed, the platform utilizes native GCP monitoring to track system health and operational metrics8. Cloud Trace provides distributed tracing for all API endpoints and agent execution steps, while BigQuery Agent Analytics tracks real-time token consumption, user journey flows, and automated evaluations, ensuring full observability into production workloads8.
Architectural Conclusions
The functional capabilities of WisdomAI—including dynamic curriculum planning, adaptive user routing, and conflict analysis—are well-designed for enterprise training needs1. However, scaling the platform for enterprise deployment requires refactoring its structural layout to align with modern web and agentic patterns1.
Reorganizing the codebase into self-contained domain directories ensures scalability and clean separation of concerns as new features are added9. Decoupling dynamic local storage from application code directories mitigates security risks, such as accidental data leaks, while preparing the platform for its planned SQL database migration1.
At the AI execution layer, moving from hardcoded Python agents to declarative Agent Skills (under .agents/skills/) simplifies prompt management and reduces context token usage11. Implementing "Luck Elimination" through native SDK Lifecycle Hooks and Policies ensures that business rules are enforced safely and consistently1. Finally, establishing automated testing, asynchronous test fixtures, and CLI-driven evaluation pipelines ensures long-term reliability, positioning WisdomAI for successful enterprise deployment5.
